## Summary

This phase focuses on establishing the `@pikacss/eslint-config` package using the repository's standard scaffolding tools. The package will be a shareable ESLint configuration designed for ESLint 9's "Flat Config" system.

The research confirms that the existing `scripts/newpkg.ts` provides the correct foundation (build with `tsdown`, test with `vitest`, `package.json` structure). The primary implementation detail specific to this package is the adoption of Flat Config patterns: exporting arrays of configuration objects and managing plugins as direct dependencies rather than peer dependencies (a key shift from legacy ESLint configs).

**Primary recommendation:** Use `pnpm newpkg` to scaffold, then configure `peerDependencies` for ESLint 9 and prepare `src/index.ts` to export a Flat Config array.

## Architecture Patterns

### Recommended Project Structure
Standard package structure provided by `newpkg.ts`:
```
packages/eslint-config/
├── src/
│   └── index.ts         # Entry point (exports config array)
├── dist/                # Build output (generated by tsdown)
├── tests/               # Unit tests
├── package.json         # Exports definition
├── tsdown.config.ts     # Build config
└── tsconfig.json        # TypeScript config
```

### Pattern 1: Flat Config Export
**What:** Exporting an array of configuration objects instead of a single object or `extends` string.
**When to use:** All ESLint 9+ configurations.
**Example:**
```typescript
// src/index.ts
import { Linter } from 'eslint'

const config: Linter.Config[] = [
  {
    ignores: ['dist', 'node_modules']
  },
  // ... other configs
]

export default config
```

### Pattern 2: Plugins as Dependencies
**What:** In Flat Config, plugins are imported directly in the config file. Therefore, they should be listed as **`dependencies`** in `package.json`, not `peerDependencies`.
**Why:** Users import your config, which imports the plugins. Node resolution handles the rest. This simplifies user setup.
**Exception:** `eslint` itself remains a `peerDependency`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| **Scaffolding** | Manual `mkdir` & `package.json` creation | `pnpm newpkg` | Ensures consistency with monorepo standards |
| **Build System** | Custom `tsc` or `rollup` config | `tsdown` | Pre-configured for dual emit and d.ts generation |
| **Config Loading** | Custom config loader | ESLint Flat Config | Native support in ESLint 9 |

## Common Pitfalls

### Pitfall 1: Peer Dependency Misunderstanding
**What goes wrong:** Listing plugins as `peerDependencies` in a Flat Config world.
**Why it happens:** Legacy ESLint configs required this because ESLint resolved plugins relative to the user's project.
**How to avoid:** List plugins (e.g., `eslint-plugin-vue`) as `dependencies` so they are installed with your config package.

### Pitfall 2: Missing `type: "module"`
**What goes wrong:** ESLint 9 prefers ESM, but package is treated as CJS.
**Why it happens:** Missing `type: "module"` in `package.json`.
**How to avoid:** Ensure `pnpm newpkg` template includes `"type": "module"` (it does).

### Pitfall 3: Build Output for Local Dev
**What goes wrong:** Changes in `src/` not reflected in consuming packages during dev.
**Why it happens:** "Hybrid Export Strategy" was rejected; package points to `dist/`.
**How to avoid:** Run `pnpm build --watch` in `packages/eslint-config` when developing.
