  <files>

  <action>
Implement API comparison following RESEARCH.md patterns:

1. Update src/types.ts with comparison result types:
   - SignatureMismatch: { apiName, file, line, extracted, documented, differences: string[] }
   - Contradiction: { apiName, message, locations: Array<{ file, line, signature }> }
   - PackageCoverage: { packageName, totalAPIs, documentedAPIs, undocumentedAPIs: string[], coveragePercent }
   - ComparisonResult: { package: PackageInfo, mismatches: SignatureMismatch[], coverage: PackageCoverage }

2. Create comparator.ts with core functions:
   - `compareSignatures(extracted: string, documented: string, context: DocumentationType): { matches: boolean, differences?: string[] }`:
     * Normalize both signatures using normalizeSignature()
     * If context is API_REFERENCE, require exact match
     * If context is GUIDE, allow simplified signatures (check core elements only: function name, required params)
     * Return detailed differences if mismatch: parameter differences, return type differences
   
   - `compareAPIs(extractedAPIs: ExtractedAPI[], documentedAPIs: DocumentedAPI[]): SignatureMismatch[]`:
     * Create map of documented APIs by name
     * For each extracted API, find matching documented API
     * Compare signatures using compareSignatures()
     * Collect all mismatches with file:line context
   
   - `calculateCoverage(packageName: string, extractedAPIs: ExtractedAPI[], documentedAPIs: DocumentedAPI[]): PackageCoverage`:
     * Create set of documented API names
     * Find undocumented APIs: extractedAPIs.filter(api => !documented.has(api.name))
     * Calculate coverage percentage: (documented / total) * 100
     * Return per-package breakdown
   
   - `detectContradictions(documentedAPIs: DocumentedAPI[]): Contradiction[]`:
     * Group APIs by name across all documentation files
     * For each API appearing in multiple files, compare signatures
     * Flag if signatures differ (after normalization)
     * Report all contradictory locations

3. Comparison heuristics (from RESEARCH.md Pattern 4):
   - Normalize whitespace, colons, arrows before comparison
   - Handle type alias expansion (check both expanded and aliased forms if possible)
   - For GUIDE context, allow parameter name differences but enforce type matching
   - Preserve detailed difference messages for debugging

4. Follow RESEARCH.md "Pattern 4: Signature Normalization & Comparison" and "Pattern 6: Per-Package Coverage Tracking"
  </action>
