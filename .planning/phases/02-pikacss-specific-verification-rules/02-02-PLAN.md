  <action>
Implement ESLint rule to validate TypeScript module augmentation patterns in plugin examples (PIKA-06 requirement).

**Expected pattern (from AGENTS.md Plugin Module Augmentation Pattern):**

```typescript
declare module '@pikacss/core' {
  interface EngineConfig {
    myPluginOption?: MyOptions
  }
  interface Shortcuts {
    myShortcut: MyShortcutDefinition
  }
}

export function myPlugin(options?: MyOptions): EnginePlugin {
  return defineEnginePlugin({
    name: 'my-plugin',
    async configureEngine(engine) {
      engine.registerShortcut('myShortcut', { ... })
    }
  })
}
```

**Implementation approach:**

1. Create `.eslint/rules/pika-module-augmentation.ts`:
   - Use `ESLintUtils.RuleCreator`
   - Detect files that export plugin functions (pattern: `defineEnginePlugin`)
   - Check for presence of `TSModuleDeclaration` with moduleReference `@pikacss/core`
   - Validate that augmentation includes interface declarations
   - Common augmentation targets: `EngineConfig`, `Shortcuts`, `Selectors`, `Preflights`

2. Detection logic:
   - Scan AST for `ExportNamedDeclaration` or `ExportDefaultDeclaration`
   - Check if export contains `defineEnginePlugin` call
   - If yes, require `TSModuleDeclaration` with literal `'@pikacss/core'`
   - Warn if augmentation missing but plugin exports configuration

3. Error messages:
   - missingAugmentation: "Plugin exports configuration but missing TypeScript module augmentation for '@pikacss/core'. Users will not get type completions."
   - suggestPattern: "Add: declare module '@pikacss/core' { interface EngineConfig { yourOption?: YourType } }"

4. Export from `.eslint/rules/index.ts`:
   ```typescript
   export { pikaModuleAugmentationRule } from './pika-module-augmentation'
   ```

5. Create test suite `.eslint/tests/rules/pika-module-augmentation.test.ts`:
   - Valid cases: plugin with augmentation, non-plugin files (no violation)
   - Invalid cases: plugin without augmentation, plugin with wrong module target
   - Use TypeScript parser configuration to handle `declare module` syntax

**Key insight from RESEARCH.md Common Pitfall 6:** Not testing module augmentation means users get missing autocomplete. This rule prevents that.

Why presence check first: Semantic validation (correct interfaces, correct types) is Phase 3 API verification work. Phase 2 focuses on structural presence.
Why plugins only: Non-plugin code doesn't need to augment @pikacss/core types.
  </action>

    - Rule correctly ignores non-plugin files (no false positives)
    - Error message suggests correct augmentation pattern

  <action>
Implement custom ESLint formatter to provide detailed, actionable error messages for PikaCSS constraint violations (per CONTEXT.md decision: detailed format with fix suggestions and docs links).

**Implementation (following RESEARCH.md Pattern 3):**

1. Create `.eslint/formatters/pikacss-detailed.ts`:
   - Implement `ESLint.Formatter['format']` function signature
   - Group errors by file (per CONTEXT.md decision)
   - For PikaCSS rules (ruleId starts with `pikacss/`), add extra context:
     - Fix suggestion from message.suggestions[0]
     - Documentation link: `https://docs.pikacss.io/verification/{ruleId}`
   - Color-coded output: red for errors, yellow for warnings, green for success summary
   - Exit with error count summary

2. Output format:
   ```
   /path/to/file.md
     12:5  ERROR  pika() argument must be statically analyzable...  pikacss/pika-build-time
       Fix: Consider using CSS variables: pika({ color: 'var(--color)' })
       Docs: https://docs.pikacss.io/verification/pikacss/pika-build-time
   
     23:10 WARN   Cannot import @pikacss/unplugin from core layer  pikacss/pika-package-boundaries
       Fix: Use @pikacss/core for core package implementations
       Docs: https://docs.pikacss.io/verification/pikacss/pika-package-boundaries
   
   ✖ 2 errors, 1 warning
   ```

3. Formatter features:
   - File path headers (group by file per CONTEXT.md)
   - Position (line:column)
   - Severity level (ERROR/WARN)
   - Message text
   - Rule ID
   - **PikaCSS enhancement:** Fix suggestion + docs link for pikacss/* rules
   - Summary line with counts

4. Usage (for documentation in SUMMARY):
   ```bash
   eslint --format .eslint/formatters/pikacss-detailed.ts .
   ```

**Follow RESEARCH.md formatter example (lines 176-210) for structure.**

Why custom formatter: Built-in formatters don't provide fix suggestions or documentation links. Custom formatter makes violations actionable.
Why file grouping: Easier to find and fix all issues in one file at a time (per CONTEXT.md decision).
Why docs links: Users can immediately understand WHY the constraint exists and HOW to fix it properly.
