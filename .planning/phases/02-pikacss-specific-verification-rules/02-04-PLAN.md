Update ESLint flat config to load and apply PikaCSS custom rules to markdown code blocks.

**Dependencies:** Uses pikaBuildTimeRule, pikaPackageBoundariesRule, pikaModuleAugmentationRule from Plan 02-01 and 02-02.

**Implementation:**

1. Import PikaCSS rules:
```javascript
import deviltea from '@deviltea/eslint-config'
import { pikaBuildTimeRule, pikaPackageBoundariesRule, pikaModuleAugmentationRule } from './.eslint/rules/index.ts'

export default await deviltea({
  stylistic: {
    overrides: {
      'style/no-mixed-spaces-and-tabs': 'warn',
    },
  },
  ignores: [
    '.planning/**',
  ],
  plugins: {
    pikacss: {
      rules: {
        'pika-build-time': pikaBuildTimeRule,
        'pika-package-boundaries': pikaPackageBoundariesRule,
        'pika-module-augmentation': pikaModuleAugmentationRule,
      },
    },
  },
  rules: {
    'pikacss/pika-build-time': 'error',
    'pikacss/pika-package-boundaries': 'error',
    'pikacss/pika-module-augmentation': 'error',
  },
})
```

2. Key configuration decisions:
   - All PikaCSS rules set to 'error' severity (blocking per CONTEXT.md for Vite/Nuxt compatibility)
   - Rules apply to all files including markdown code blocks (handled by @deviltea/eslint-config)
   - `.planning/**` remains ignored (planning docs separate from project docs)

**Why plugin registration:** ESLint flat config requires explicit plugin object with rules before they can be used in rules configuration.
**Why error severity:** Build-time constraint and package boundaries are architectural requirements, not style preferences. Violations break user projects.

Note: Custom formatter will be specified at CLI invocation time, not in config (allows flexibility for different output formats).

  <name>Task 2: Create unified PikaCSS validation script with custom formatter</name>
  <files>
    scripts/run-pikacss-validation.sh
    package.json
  </files>
  <action>
Create bash script that orchestrates PikaCSS-specific validation: ESLint with custom rules and formatter, plus integration tests.

**Dependencies:** 
- Uses ESLint rules from Plan 02-01 and 02-02 (imported via eslint.config.mjs from Task 1)
- Uses custom formatter from Plan 02-02 (.eslint/formatters/pikacss-detailed.ts)
- Invokes integration tests from Plan 02-03 (pnpm test:eslint)

**Create scripts/run-pikacss-validation.sh:**

```bash
#!/usr/bin/env bash
set -euo pipefail

# Colors
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

echo -e "${GREEN}=== PikaCSS Documentation Validation ===${NC}\n"

FAILED=0

# Step 1: ESLint with PikaCSS rules and custom formatter
echo -e "${YELLOW}[1/2] Running ESLint with PikaCSS custom rules...${NC}"
if pnpm eslint --format ./.eslint/formatters/pikacss-detailed.ts \
    "docs/**/*.md" \
    "packages/*/README.md" \
    "AGENTS.md" \
    ".github/skills/**/*.md"; then
  echo -e "${GREEN}✓ ESLint validation passed${NC}\n"
else
  echo -e "${RED}✗ ESLint validation failed${NC}\n"
  FAILED=$((FAILED + 1))
fi

# Step 2: Multi-bundler integration tests
echo -e "${YELLOW}[2/2] Running multi-bundler integration tests...${NC}"
if pnpm test:eslint; then
  echo -e "${GREEN}✓ Integration tests passed${NC}\n"
else
  echo -e "${RED}✗ Integration tests failed${NC}\n"
  FAILED=$((FAILED + 1))
fi

# Summary
echo -e "${GREEN}=== Validation Summary ===${NC}"
if [ $FAILED -eq 0 ]; then
  echo -e "${GREEN}All PikaCSS validations passed ✓${NC}"
  exit 0
else
  echo -e "${RED}$FAILED validation step(s) failed ✗${NC}"
  exit 1
fi
```

**Make executable:**
```bash
chmod +x scripts/run-pikacss-validation.sh
```

**Add npm script to package.json:**
```json
{
  "scripts": {
    "validate:pikacss": "./scripts/run-pikacss-validation.sh"
  }
}
```

**Script features (following scripts/run-all-checks.sh pattern from Phase 1):**
- Color-coded output for readability
- Runs both ESLint (with custom formatter) and integration tests
- Continues on first failure to show all issues
- Exit code reflects overall success/failure (CI-friendly)
- Clear summary at end

**Why separate from run-all-checks.sh:** Phase 1 script handles structural validation (links, file refs, placeholders). This script handles PikaCSS-specific constraints. Both can run independently or together.

**Update .github/workflows/docs-validation.yml:**

Read current workflow, then add new job after existing structural validation:

```yaml
name: Documentation Validation

on:
  push:
    branches: [main]
    paths:
      - 'docs/**/*.md'
      - 'packages/*/README.md'
      - 'AGENTS.md'
      - '.github/skills/**/*.md'
      - '.eslint/**'
      - 'scripts/run-*.sh'
  pull_request:
    paths:
      - 'docs/**/*.md'
      - 'packages/*/README.md'
      - 'AGENTS.md'
      - '.github/skills/**/*.md'
      - '.eslint/**'
      - 'scripts/run-*.sh'

jobs:
  structural-validation:
    name: Structural Validation
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - uses: pnpm/action-setup@v4
        with:
          version: 10
      
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      
      - name: Install dependencies
        run: pnpm install --frozen-lockfile
      
      - name: Run structural validation
        run: ./scripts/run-all-checks.sh

  pikacss-validation:
    name: PikaCSS Constraint Validation
    runs-on: ubuntu-latest
    needs: structural-validation  # Run after structural checks pass
    steps:
      - uses: actions/checkout@v4
      
      - uses: pnpm/action-setup@v4
        with:
          version: 10
      
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      
      - name: Install dependencies
        run: pnpm install --frozen-lockfile
      
      - name: Build packages (needed for integration tests)
        run: pnpm build
      
      - name: Run PikaCSS validation
        run: pnpm validate:pikacss
      
      - name: Upload test results on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: pikacss-validation-results
          path: |
            .eslint/tests/integration/
            /tmp/pika-*/
          retention-days: 7
```

**Key CI workflow decisions:**
- **Sequential execution:** `needs: structural-validation` ensures structural checks pass before running heavier PikaCSS validation (saves CI time on basic errors)
- **Build step required:** Integration tests need built packages to test against
- **Artifact upload on failure:** Captures test output and temp directories for debugging
- **Trigger paths:** Runs on changes to docs, ESLint config, or validation scripts
- **Environment:** Ubuntu with Node 20, pnpm 10 (matches project standards)

**Why after structural:** No point running bundler tests if markdown syntax is broken. Sequential saves ~5 min CI time per broken commit.
**Why build packages:** Integration tests install packages via `workspace:*`, needs dist/ to exist.
