  <action>
Implement ESLint rule to detect runtime-dynamic arguments in `pika()` calls (PIKA-01 requirement).

**Implementation approach (following RESEARCH.md Pattern 1):**

1. Create `.eslint/rules/pika-build-time.ts`:
   - Use `ESLintUtils.RuleCreator` with docs URL pattern
   - Detect `CallExpression` nodes where `callee.name === 'pika'`
   - Use `getParserServices()` to access TypeScript type checker
   - Check if argument types are literal/const or runtime variables
   - Report violations with messageId 'runtimeArg' and suggestion 'suggestCssVar'

2. Key detection logic:
   - Static literals: `pika({ color: 'red' })` → VALID
   - Const references: `const C = 'blue'; pika({ color: C })` → VALID
   - Runtime variables: `function f(c) { pika({ color: c }) }` → INVALID
   - Use TypeScript type flags to distinguish: `type.flags & ts.TypeFlags.Literal`

3. Error messages:
   - runtimeArg: "pika() argument must be statically analyzable at build time. Found runtime variable: {{ varName }}"
   - suggestCssVar: "Consider using CSS variables: pika({ color: 'var(--color)' })"

4. Export rule from `.eslint/rules/index.ts`:
   ```typescript
   export { pikaBuildTimeRule } from './pika-build-time'
   ```

5. Create test suite `.eslint/tests/rules/pika-build-time.test.ts`:
   - Use `RuleTester` with TypeScript parser configuration
   - Valid cases: literals, const references, template literals with static parts
   - Invalid cases: function parameters, props access, let variables
   - Verify error messages and suggestions match

**Follow RESEARCH.md example code structure for type-aware rules (lines 323-373).**

Why AST analysis: Regex cannot handle all JavaScript/TypeScript patterns. Parser services provide type information that's critical for distinguishing static vs runtime.
Why suggestions: Guide users toward correct pattern (CSS variables) rather than just blocking.

    - Test file has both valid and invalid test cases (minimum 3 each)

    - Error messages include variable name and CSS variable suggestion
  </verify>

  <action>
Implement ESLint rule to enforce monorepo layer architecture (PIKA-02 requirement).

**Layer order (from AGENTS.md and RESEARCH.md Pattern 4):**
```typescript
const LAYER_ORDER = ['core', 'integration', 'unplugin', 'framework'] as const;
const PACKAGE_LAYERS: Record<string, typeof LAYER_ORDER[number]> = {
  '@pikacss/core': 'core',
  '@pikacss/integration': 'integration',
  '@pikacss/unplugin-pikacss': 'unplugin',
  '@pikacss/vite-plugin-pikacss': 'framework',
  '@pikacss/nuxt-pikacss': 'framework',
};
```

**Implementation:**

1. Create `.eslint/rules/pika-package-boundaries.ts`:
   - Use `ESLintUtils.RuleCreator`
   - Detect `ImportDeclaration` nodes
   - Determine current package layer from `context.filename` (parse path to extract package name)
   - Determine imported package layer from `node.source.value`
   - Validate: current layer can only import from same or lower layer
   - Report violations with clear explanation of violated boundary

2. Validation logic:
   - Core can import: only core (no dependencies per AGENTS.md)
   - Integration can import: core, integration
   - Unplugin can import: core, integration, unplugin
   - Framework can import: any layer

3. Error message:
   - invalidImport: "Cannot import {{ imported }} ({{ importedLayer }}) from {{ importer }} ({{ importerLayer }}). Violates layer boundary: {{ expectedOrder }}"

4. Export from `.eslint/rules/index.ts`:
   ```typescript
   export { pikaPackageBoundariesRule } from './pika-package-boundaries'
   ```

5. Create test suite `.eslint/tests/rules/pika-package-boundaries.test.ts`:
   - Valid cases: core imports nothing, integration imports core, framework imports anything
   - Invalid cases: core imports integration (upward), integration imports unplugin (skip layer)
   - Use different filenames to simulate different package contexts

**Follow RESEARCH.md Pattern 4 structure (lines 218-257).**

Why layer enforcement: Prevents circular dependencies and maintains clean architecture documented in AGENTS.md.
Why in ESLint: Catches violations in documentation examples before they reach production.
  </action>

    - Test file covers all layer combinations (minimum 6 test cases)
    - Rule correctly blocks upward imports (core → integration)
    - Rule allows downward imports (framework → core)
    - Error messages clearly state which layer boundary was violated
  </verify>
